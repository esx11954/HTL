## 0. はじめに：LPIC-1とは何か

### LPIC-1試験の正体

LPIC-1は **Linux Professional Institute** が認定する、Linuxシステム管理者向けの資格試験です。

| 項目 | 内容 |
|---|---|
| 試験回数 | **2回**（101試験 + 102試験） |
| 合格基準 | 各試験で **65%程度**（500点満点中約500点で合格） |
| 問題数 | 1試験あたり **60問** |
| 試験時間 | **90分** |
| 過去問流用率 | **高い**（暗記で対応可能） |

LPIC-1は「Linuxを業務で扱う最低限の知識を持っています」と証明できる資格です。SIerやインフラ系の現場で配属の条件になっていたり、評価対象になっていたりします。

「合格に必要なのは仕組みの完全理解じゃなくて、過去問の答えを覚えること」が現実です。難しい仕組みは追わず、「**シャープに試験だけ見る**」スタンスで進めましょう。

### 第1回で扱う「第1章 システムアーキテクチャ」とは

「Linuxが動くコンピュータの仕組み」を理解する章です。4つの大テーマで構成されます：

1. **ハードウェアとデバイス管理**（CPU、メモリ、USB、デバイスドライバ）
2. **電源ONから起動完了までの流れ**（BIOS → ブートローダ → カーネル → init）
3. **SysVinit**（昔の起動の仕組み、ランレベル）
4. **systemd**（今の起動の仕組み、Unit、systemctl）

LPIC-1試験全体の中で **約15%** がこの章から出題されます。
1回で集中して攻める価値のある範囲です。

## 1.1 ハードウェアの基本知識と設定

### 1.1.1 基本的なシステムハードウェア

#### ここで学ぶこと

「Linuxが動くコンピュータって、中に何が入ってるの？」という超基本の話です。

「ハードウェアの話は知ってるよ」と思う方も読んでください。LPIC試験では「これは何のためにある部品ですか?」「この部品の特徴を答えなさい」というレベルで問われるので、**普段なんとなく知っていること** を **言葉で説明できるレベル** まで上げるのがこのセクションの目的です。

このセクションが終わると、次のことができるようになります：

- PCの主要部品を5つ以上、名前と役割で言える
- 「なぜメモリが必要なのか」を初めて聞いた人に説明できる
- HDDとSSDの違いを2つ以上挙げられる
- 試験で出題される「アーキテクチャ」「ホットプラグ」「揮発性」といった専門用語の意味が分かる

#### まずは身近な話から ─ PCの中身

あなたが今この資料を読んでいるPC(またはスマホ)も、中身は同じ部品で構成されています。

PCの蓋を開けたことがある人なら見たことがあるはずですが、ない人のために言うと、PCの中身は大体こんな構成です：

![PCの中身 - 一般的な構成](images/01_pc_components.png)

中央には **マザーボード** という大きな基板があり、その上に CPU・メモリ・ストレージ・拡張カードといった主要部品が**乗っかっている**、というのが基本構造です。

> 💡 **マザーボード(母なる基板)** という名前の通り、すべての部品をつなぐ「土台」の役割をしています。マザーボードがなければ、CPUもメモリもただの板きれで、お互いに会話できません。マザーボード上に張り巡らされた電気回路を通じて、部品同士がデータをやり取りしています。

LPIC試験では、これら一つ一つの**正式名称・役割・特徴**を覚える必要があります。「知ってるよ」と思っても、実は試験で正確に答えるには細かい点を押さえる必要があるので、丁寧に見ていきましょう。

#### CPU（シーピーユー）

##### 何の部品か

**コンピュータの「頭脳」** です。あらゆる計算・処理を担当する、文字通り中心的なパーツ。

正式名称は **Central Processing Unit（中央処理装置）** で、頭文字を取って CPU と呼ばれます。

CPUは小さな四角いチップで、マザーボードの中央あたりにソケットと呼ばれる差し込み口に挿さっています。普段は **CPUクーラー**（冷却装置）に覆われているので、PCを開けてもCPU本体は見えないことが多いです（CPUは高速に動作するので熱を持ち、冷やさないと壊れるため）。

##### CPUは何をしているか

ものすごくシンプルに言うと、CPUは「**メモリに置かれたプログラム(命令の塊)を、上から順に読んで実行する**」という仕事だけをしています。

たとえば、あなたがブラウザでYouTubeを見ているとき、裏では：

1. メモリ上に置かれたChromeのプログラムを CPU が1命令ずつ読む
2. 「画面のここに動画を描け」「ネットからデータを受け取れ」といった命令を実行する
3. これを **1秒間に何十億回** も繰り返している

この「**1秒間に何回処理できるか**」の指標が **動作周波数（クロック数）** で、単位は **GHz（ギガヘルツ）** です。

> **1GHz = 1秒間に10億サイクル**

例: `Intel Core i7-3630QM CPU @ 2.40GHz` というCPUは、1秒間に約24億サイクルの処理ができる、ということです。

##### CPUの種類を表す「アーキテクチャ」

CPUには実はいろいろな**設計の流派**があります。これを **アーキテクチャ** と呼びます。

> **アーキテクチャ** = 建築でいう「日本建築」「西洋建築」みたいなもの。基本的な設計思想の違いを指す言葉です。
> ハードウェア用語としては特に **CPUの種別** を指して使うことが多く、試験ではこの意味で出てきます。

なぜアーキテクチャの違いが重要かというと、**アーキテクチャが違うと同じプログラムが動かない** からです。

LPIC試験で覚えるべき主要アーキテクチャは以下の3つです：

| 名前 | どんなCPUか | 身近な例 |
|---|---|---|
| **x86** / **IA-32** | Intelが作った32ビット設計。PCで最も普及した | 古めのWindows PC |
| **x86-64** | x86を64ビットに拡張したもの | **現在のPCはほぼこれ**（AMD64とも呼ぶ） |
| **ARM** | スマホや組み込み機器向けの省電力設計 | iPhone、Android、Raspberry Pi、Mac M1/M2 |

> 💡 **なぜ覚える必要があるか**: Linuxは「どのCPUアーキテクチャでも動くOS」が大きな売りの1つです。試験では「Linuxは多くのアーキテクチャに対応している」という文脈で出題されます。

##### 🎯 試験ではこう問われる

- 「CPUの動作周波数の単位は?」 → **GHz**
- 「x86, x86-64, ARM のような区分けを何という?」 → **アーキテクチャ**
- 「Linuxが対応するCPUは?」 → **多くのアーキテクチャに対応**

#### メモリ（メインメモリ・主記憶）

##### 何の部品か

CPUが作業するときの **「机の上」** にあたる部品です。マザーボード上のCPUの隣に、細長い基板が何枚か挿さっているのを見たことがある人もいるかもしれません。それがメモリです。

正式には **Random Access Memory（RAM）** と呼びます。

##### なぜ必要か（HDDがあるのに、なぜ別途メモリがいるのか）

「保存するならハードディスクがあるじゃん。なぜわざわざメモリが必要なの?」と思うかもしれません。とても良い疑問です。

答えは **速度の問題** です。CPUの動作速度はGHz単位（10億サイクル/秒）と凄まじく速いのに対し、ハードディスク（特にHDD）の読み書きはミリ秒（1/1000秒）単位とかなり遅い。CPUがHDDからの応答を毎回待っていたら、計算が全く進みません。

これを「机の上で書類を広げて仕事をする人」に例えると、こうなります：

![速度の階層](images/02_speed_hierarchy.png)

倉庫（HDD）から書類を取りに行くのに毎回時間がかかるので、「**今扱っている書類だけは机の上（メモリ）に広げておく**」というルールにしているわけです。CPUは机の上のものなら超高速に読めるので、これで効率よく仕事ができます。

実際の動きはこうです：

1. あなたがアプリ（例: Chrome）を起動する
2. OSが HDD/SSD に保存されている Chrome のプログラムを読み込み、**メモリにコピー** する
3. CPU はメモリ上の Chrome を実行する
4. Chrome を閉じると、メモリ上のChromeのデータは消える（HDD上のファイル本体は残る）

つまり「メモリ = HDDから今使うものをコピーしてきた一時的な作業エリア」 という関係です。

##### メモリの致命的な特徴：電源を切ると消える

メモリは **電気が流れている間しかデータを保持できません**。

これを **揮発性（きはつせい）** と言います。蒸発するイメージで覚えてください。

つまり：
- PCの電源を切る → **メモリの中身は全部消える**
- 大事なファイルは HDD/SSD に保存しないと、シャットダウンしたら無くなる

普段あなたがPCを使っていて「保存しないとファイルが消える」と注意されるのはこれが理由です。「**保存**」とは実は「メモリからHDDへ書き写す操作」のことなんです。

> 💡 **揮発性 vs 不揮発性**: 試験では「揮発性メモリ = メモリ(RAM)」「不揮発性ストレージ = HDD/SSD」と対比で問われることがあります。

##### Linuxを動かすのに必要なメモリ量

教科書的には **512MB〜1GB以上** とされていますが、現実的にはサーバ用途なら **最低1GB、できれば4GB以上** ほしいところです。

ちなみに「**ROM**（Read Only Memory）」という別の種類のメモリもあり、こちらは **読み出し専用** で **電源を切っても内容が消えません**。BIOSプログラムはこのROM（厳密にはフラッシュROM）に書き込まれています。詳しくは次の1.1.2で扱います。

##### 🎯 試験ではこう問われる

- 「メモリの内容は電源を切るとどうなる?」 → **消える(揮発性)**
- 「Linuxが動作するのに必要なメモリ量は?」 → **512MB〜1GB以上**
- 「読み出し専用で電源OFFでも消えないメモリは?」 → **ROM**

#### ハードディスク（HDD・SSD）

##### 何の部品か

データを **長期保存する** ための部品です。「**倉庫**」のイメージ。電源を切っても中身が残ります（**不揮発性**）。これがメモリとの最大の違いです。

容量は数百GB〜数TBが一般的で、メモリ（数GB〜数十GB）よりはるかに大きな容量を持ちます。代わりに速度はメモリより遅いです。

ハードディスクと総称される中には、内部の仕組みが全く違う **HDD** と **SSD** の2タイプがあります。

##### HDDとSSDの違い

![HDD vs SSD](images/03_hdd_vs_ssd.png)

最近のPCでは「ストレージはSSDですか?HDDですか?」と聞かれることがあります。これは速度や価格に直結するからです。

> 💡 **フラッシュメモリ**: USBメモリやSDカードと同じ仕組みの記憶素子です。電源を切っても内容が消えないタイプの半導体メモリ。SSDはこれを内部に大量に並べて作られています。

最近のPCはほとんどSSDになっていますが、外付けの大容量ストレージ（写真や動画のバックアップ用）には今でもHDDがよく使われます。容量あたりの価格が安いからです。

##### 試験で覚えるべきポイント

LPIC試験では、**HDDとSSDの細かい違いは問われません**。次のことを押さえておけばOKです：

- どちらも「**ハードディスク**」「**ストレージ**」と総称される
- 電源を切っても **データが消えない**（**不揮発性**）
- メモリより **遅いが、容量が大きい**
- HDDとSSDの区別は「**HDD=磁気円盤、SSD=フラッシュメモリ**」とだけ覚える

##### 🎯 試験ではこう問われる

- 「電源を切っても消えないのはどっち?」 → **HDD/SSD**（メモリではない）
- 「Linuxの動作にHDDは必須?」 → **必須ではない**（ただし通常は使う）
- 「メモリの代わりに長期保存に使う部品は?」 → **HDD/SSD（ストレージ）**

#### 入力装置

キーボード・マウスのこと。あなたがPCに情報を「入力」する装置だから「入力装置」です。

LPIC試験で押さえるべきポイントは1つだけです：

> サーバ機では **キーボード・マウスが接続されていないことが多い**

なぜか? サーバは普段「サーバルーム」という別室にラック収納されていて、人が直接触らない設計だからです。サーバはネットワーク越しに **SSH** という仕組みでリモートからアクセスして操作します（SSHは102試験で詳しく扱います）。だからキーボードもマウスもモニタも繋がっていません。電源と LANケーブルだけが繋がっている、というのが典型的なサーバの姿です。

ただし機種によっては、**「キーボードが繋がってないとエラーになって起動できない」** という昔ながらの仕様のマザーボードもあります。その場合は **BIOS設定で「キーボードなしOK」** と教えてあげる必要があります（次の1.1.2で出てくるBIOS設定の話とつながります）。

#### 拡張カード

##### 何の部品か

PCの基本機能（CPU、メモリ、ストレージ）以外の **追加機能** を提供するカード型の部品です。マザーボードの「拡張スロット」と呼ばれる差込口にカチッと挿して使います。

イメージとしては「**後付けで機能を増やせるカートリッジ**」みたいなものです。

##### 主要な拡張カード（試験頻出）

| 名前 | 何ができる |
|---|---|
| **NIC**（Network Interface Card） | **ネットワーク（LAN）に接続**できるようにする ← 最重要 |
| サウンドカード | 高音質な音を出す |
| ビデオカード（GPU） | ゲームや動画編集の処理を高速化 |
| SCSIカード | 古い規格のストレージ接続 |
| RAIDカード | 複数のHDDを束ねて冗長化 |

> 💡 **NICだけは絶対に覚えてください**。「Network Interface Card」の略で、有線LANポートを提供するカードです。LPICではネットワーク関連の話題で何度も登場します。サーバが外部と通信できる根本がここなので、「NIC=ネットワーク」を反射的に答えられるようにしましょう。

##### 注意事項

拡張カードを抜き差しするときは、**必ず電源を切ってから** 行います。電源ONのまま挿抜するとショートしてカードが壊れる可能性があります。

これに対して、次に出てくる **USB機器は電源ONのまま抜き差しできる** のが大きな違いです。この対比は試験で問われることがあります。

#### USB機器

##### 何が嬉しい規格か

**USB（Universal Serial Bus）** は、**1種類のコネクタでいろんな機器を繋げる** ように設計された接続規格です。

「**Universal**（汎用）」の名前の通り、USBが普及する以前はマウス用・キーボード用・プリンタ用・外付けHDD用で全部違うコネクタを使っていました。それが「全部この口でOK」になったことで、PCの後ろの接続ポートが一気にシンプルになりました。

##### USBの特徴（試験頻出）

これは試験でそのまま問われるので、太字部分はしっかり覚えてください：

- **最大127台** までのUSBデバイスを接続可能
- 同一のコネクタでさまざまなデバイスを接続可能
- **電源ONのまま接続・取り外し可能**（これを **ホットプラグ** と呼ぶ）
- **プラグ＆プレイ** に対応（挿せば自動認識）
- USBポートから機器に **電源供給可能**（スマホの充電がこれ）

> 💡 **ホットプラグ** = "Hot"（電源が入った熱い状態）で "Plug"（挿す）できる、という意味。
> 拡張カードは電源を切らないと挿せないので「コールドプラグ」、USBは「ホットプラグ」、と対比で覚えると良いです。

##### USBのバージョン（覚える）

USBにはバージョンがあり、それぞれ最大データ転送速度が異なります：

| バージョン | 最大データ転送速度 |
|---|---|
| USB 1.0 / 1.1 | **12 Mbps**（メガビット/秒） |
| USB 2.0 | **480 Mbps** |
| USB 3.0 | **5 Gbps**（ギガビット/秒） |
| USB 3.1 | **10 Gbps** |

> ⚠ **速度の単位に注意**: 1.0/1.1/2.0は **メガ**（Mbps）、3.0以降は **ギガ**（Gbps）です。**桁が3桁違います**。
> 試験で「USB 2.0の転送速度は?」と聞かれて「5Gbps」と答えると不正解。「480Mbps」が正解です。

##### 🎯 試験ではこう問われる

- 「USBデバイスは何台まで接続可能?」 → **127台**
- 「電源ONのまま機器を接続できる仕組みを何と呼ぶ?」 → **ホットプラグ**
- 「USB 2.0の最大データ転送速度は?」 → **480 Mbps**
- 「USBの自動認識を表す言葉は?」 → **プラグ＆プレイ**

#### 📝 1.1.1 のまとめ

| 部品 | 一言で言うと | 試験頻出の特徴 |
|---|---|---|
| **CPU** | 計算する頭脳 | GHzで速度、アーキテクチャで種類（x86/x86-64/ARM） |
| **メモリ（RAM）** | 高速だが消える作業場 | 揮発性、512MB〜1GB必要、CPUの隣 |
| **HDD / SSD** | 不揮発な倉庫 | 電源OFFでも残る、SSDはフラッシュメモリ |
| **入力装置** | キーボード・マウス | サーバでは無いことも（BIOS設定で対応） |
| **拡張カード** | 後付け機能カード | NIC=ネットワーク、電源OFFで抜き差し（コールドプラグ） |
| **USB機器** | 汎用接続規格 | 127台、ホットプラグ、速度バージョン別 |

### 1.1.2 BIOS / UEFI

#### ここで学ぶこと

PCの電源を入れた直後、**OSが起動するまでの「OSじゃない誰か」** が動いています。それが **BIOS** や **UEFI** です。

「OSの前に動く何か」とは何者か、なぜ存在するのか、何をしているのか、を理解するのがこのセクションの目的です。Linuxの起動の話（1.2）に直結する重要な前提知識です。

#### 「電源直後に動くプログラム」って何?

普段あなたがPCの電源ボタンを押すと、しばらく真っ黒な画面のあとに、メーカーロゴが出て、その後Windowsの起動画面が出てきますよね。あの「真っ黒な画面〜メーカーロゴ」のあたりで動いているのが **BIOS（バイオス）** または **UEFI（ユーイーエフアイ）** です。

OSはまだ動いていません。なぜならOSはHDD/SSDの中に眠っていて、誰かがそれを起こしに行かないと動けないからです。その「誰か」がBIOS/UEFIの仕事の1つです。

![BIOS / UEFI はどこに住んでいるか](images/04_bios_location.png)

> 💡 **電源直後の世界はカラッポ**: PCに電源が入った瞬間、メモリは空、HDDの中身もまだ読まれていない、OSも動いていません。「何かが最初に動き出さないと、何も始まらない」状態です。だから「マザーボード自体に焼き込まれた何か」が必要で、それがBIOS/UEFIなのです。

#### BIOSとは

**BIOS（Basic Input Output System：入出力基本システム）**
キーボードやハードディスクなどのデバイスを制御する、最も基本的な制御プログラムです。

- マザーボードや拡張カードに搭載された **フラッシュROM** に書き込まれている
- 電源を入れると **最初に動く**

> 💡 **フラッシュROM**：通常のROMは「読み出し専用」で書き換え不可ですが、フラッシュROMは **書き換え可能** です。だからメーカーが公開するBIOSアップデートで内容を書き換えることができます。

#### BIOSの3つの役割

BIOS/UEFIの仕事は試験でほぼ確実に問われます。次の3つを丸暗記してください：

![BIOSの3つの役割](images/05_bios_roles.png)

1. **OSを起動するためのプログラム（ブートローダ）をディスクから読み込んで実行する** ← 最重要
2. **デバイスの動作を設定する** ← セットアップ画面で行う
3. **基本的な入出力を制御する** ← OS起動前の画面表示やキー入力

#### UEFI（BIOSの後継）

**UEFI（Unified Extensible Firmware Interface）**
現在普及している、BIOSの後継ファームウェア規格です。今のPCはほぼUEFIになっています。

![BIOS vs UEFI](images/06_bios_vs_uefi.png)

UEFIの主な改善点：

| 項目 | BIOS | UEFI |
|---|---|---|
| 起動ドライブ容量制限 | 2TBまで | **制限なし**（実質TB級OK） |
| パーティション形式 | MBR | **GPT**（128パーティションまで） |
| セットアップ画面 | テキストベース | **GUI**（マウス操作可） |

> 💡 旧BIOSとUEFIをまとめて「BIOS」と呼ぶこともあります。試験ではこの呼称の揺れも頭に入れておきましょう。

#### BIOS/UEFIセットアップで設定できること

電源直後にBIOS/UEFIが画面を出している間、特定のキー（**Delete・F1・F2**など）を押すとセットアップ画面が出ます。ここで以下のような設定ができます：

- 日付や時刻（**ハードウェアクロック**）
- ディスクドライブや各種デバイスのパラメータ
- キーボードの使用 / 不使用
- 電源管理
- **起動ドライブの順序** ← 特に重要
- デバイスへのIRQ（割り込み要求）の割り当て
- 各種デバイスの使用 / 不使用

> 💡 **起動ドライブの順序とは**: 「最初に起動デバイスとして探すドライブはどれか」の設定です。普段はHDD/SSDが最初ですが、Linuxのインストール時はUSBメモリやDVDを先にすることで、そこからインストーラを起動できます。

#### POST（Power On Self Test）

電源投入時にBIOS/UEFIが行う **ハードウェア自己診断** のこと。メモリやデバイスをチェックして、問題があればビープ音やエラーメッセージで知らせます。

「POST」という言葉は試験でそのまま出るので覚えてください。

#### 📝 1.1.2 のまとめ ─ 🎯 試験ポイント

| 問われ方 | 答え |
|---|---|
| システム構成変更はどこで? | **BIOS/UEFIセットアップ画面** |
| 周辺機器のON/OFFや起動ドライブ順序は? | **BIOS/UEFI** |
| 電源投入時のハード自己診断は? | **POST** |
| BIOSが書き込まれているのは? | **フラッシュROM** |
| BIOSの後継規格は? | **UEFI** |

### 1.1.3 デバイス情報の確認

#### ここで学ぶこと

Linuxシステム上で「今このPCにどんなハードが繋がっているか」を確認する方法を学びます。`lspci`, `lsusb` のような **「ls から始まるコマンド」のシリーズ** がメインです。試験で必ず出題される範囲です。

#### Linuxの「ハードもファイル扱い」の哲学

Linuxの大きな特徴のひとつに、「**Everything is a file**（すべてはファイル）」という設計思想があります。これは、ハードウェアもファイルの形でアクセスできるようにする、という考え方です。

具体的には2つの場所が用意されています：

![Linuxの「ハードウェアをファイルとして扱う」考え方](images/07_dev_proc.png)

##### /dev ディレクトリ ─ デバイスファイル

`/dev/` 以下には、ハードウェアに対応する **デバイスファイル** が並んでいます。たとえば：

- `/dev/sda` → 1台目のHDD全体
- `/dev/sda1` → 1台目のHDDの第1パーティション
- `/dev/tty1` → 1番目の仮想コンソール
- `/dev/null` → 「捨て場」（書き込んだものは全部消える特殊なファイル）

これらのファイルに対して読み書きすると、対応するハードウェアにアクセスできます。

> 💡 これらのデバイスファイルは、後で出てくる **udev** という仕組みによって自動的に作成されます。

##### /proc ディレクトリ ─ カーネル情報

`/proc/` 以下には、**カーネルが認識しているデバイスやシステムの情報** がファイルの形で並んでいます。

| ファイル名 | 説明 |
|---|---|
| `/proc/cpuinfo` | **CPU情報** ← 試験頻出 |
| `/proc/meminfo` | **メモリ情報** |
| `/proc/interrupts` | IRQ情報 |
| `/proc/ioports` | I/Oアドレス情報 |
| `/proc/bus/usb/*` | USBデバイス情報 |
| `/proc/bus/pci/*` | PCIデバイス情報 |

これらは **仮想的なファイル**（実体がない）で、cat等で読み出した瞬間にカーネルが動的に内容を生成します。

```bash
$ cat /proc/cpuinfo
processor   : 0
vendor_id   : GenuineIntel
cpu family  : 6
model name  : Intel(R) Core(TM) i7-3630QM CPU @ 2.40GHz
（以下省略）
```

#### デバイス情報を表示するコマンド群

`/proc/` を直接読むより便利な、デバイス情報表示専用のコマンドが揃っています：

![ls* シリーズ ─ ハードウェア情報を見るコマンド群](images/08_ls_commands.png)

| コマンド | 用途 |
|---|---|
| **`lspci`** | **PCIデバイス** の情報を表示 |
| **`lsusb`** | **USBデバイス** の情報を表示 |
| `lscpu` | CPU情報を表示（`/proc/cpuinfo` より見やすい） |
| `lsblk` | ブロックデバイス（HDD、SSD等）の情報を表示 |
| `lsmod` | ロードされているカーネルモジュールを表示 |
| `dmesg` | カーネルが出力したメッセージを表示 |

> 💡 **覚え方：`ls*` シリーズ** = 「list = 一覧」の意味。`ls + 対象の略称` で覚える：
> - `pci` → lspci
> - `usb` → lsusb
> - `cpu` → lscpu
> - `blk`(block) → lsblk
> - `mod`(module) → lsmod

#### lspciコマンドの詳細

```bash
$ lspci

00:01.0 ISA bridge: Intel Corporation 82371SB...
00:02.0 VGA compatible controller: ...

```

| オプション | 効果 |
|---|---|
| `-v` | 詳細表示 |
| `-vv` | さらに詳細表示 |

| 問われ方 | 答え |
|---|---|
| ハードウェア情報を確認できるディレクトリ | **/proc** |
| CPU情報のファイル | **/proc/cpuinfo** |
| PCIデバイス情報を表示するコマンド | **lspci** |
| PCIデバイス情報を詳細表示する | **lspci -vv** |
| デバイスファイルがあるディレクトリ | **/dev** |

### 1.1.5 udev

#### ここで学ぶこと

USBメモリを挿した瞬間に `/dev/sdb1` のようなデバイスファイルが自動的に出現する仕組みを学びます。これが **udev** です。

#### なぜudevが必要か（昔の話）

`/dev` ディレクトリのデバイスファイルは、もともとはシステム管理者が **手動で大量に作成しておく必要** がありました。

例：
- 「あとで何か接続するかも」と思って `/dev/sda` から `/dev/sdz` まで全部用意
- 仮想コンソール用に `/dev/tty1` から `/dev/tty12` まで用意
- USB機器用にあれもこれも...

結果：使わないファイルだらけで、しかも事前に用意していなかった機器は使えないという問題がありました。

#### udevの仕組み ─ 自動でデバイスファイルを作る

**udev（Userspace DEVice management）** は、USB機器などが接続された瞬間に、対応するデバイスファイルを `/dev` 以下に **自動作成** する仕組みです。

![udev ─ デバイスファイルを自動作成する仕組み](images/10_udev_flow.png)

4つのステップで動いています：

1. **デバイス接続** → USB機器が物理的に挿される
2. **/sys に情報書込** → カーネルがデバイス情報を `/sys` 以下に書き出す（sysfs）
3. **udevd が判断** → udevデーモンが `/sys` を監視し、設定ファイル `/etc/udev/rules.d/` のルールに従って命名規則を決定
4. **/dev にファイル作成** → 適切な名前のデバイスファイルが `/dev/` 以下に生成される

設定ファイルを編集することで、たとえば特定のUSBメモリを必ず `/dev/usbmemory` という名前にする、といったカスタマイズも可能です。

> 💡 **/sys ディレクトリ（sysfs）**：`/proc`（procfs）と同様に、ファイルの形で情報を提供する仮想ファイルシステムです。デバイス情報がここに集約されています。
>
> 💡 **D-Bus（Desktop Bus）**：デバイス情報をアプリケーションに伝える機構。試験ではこの用語が出ることがあります。

#### 🎯 試験ポイント

- 「デバイスファイルを自動作成する仕組みは?」 → **udev**
- 「udev のデーモンプロセス名は?」 → **udevd**
- 「udevのルール設定ファイルがあるディレクトリは?」 → `/etc/udev/rules.d/`

### 1.1.6 デバイスドライバのロード

#### ここで学ぶこと

USB機器を挿しても、それを動かす **ソフト** がカーネルに入っていないと使えません。そのソフトが **デバイスドライバ** で、カーネルに取り込むことを「**ロード**」と呼びます。ここでは4つの操作コマンドを覚えます。

#### デバイスドライバ = カーネルモジュール

Linuxでは、デバイスドライバは **カーネルの一部** として用意されます。カーネル本体に最初から組み込むこともできますが、通常は **カーネルモジュール** という後付けで読み込める部品として提供されます。

> 💡 「モジュール」 = 「部品」のイメージ。必要なときだけカーネルに組み込めば、メモリも節約できる。

#### カーネルモジュール操作の4コマンド

![カーネルモジュール操作 ─ 4 つのコマンドで完結](images/11_kernel_modules.png)

「**見る → 調べる → 入れる → 抜く**」の順で覚えると、自然に流れが頭に入ります。

| コマンド | 用途 | 覚え方 |
|---|---|---|
| **`lsmod`** | ロードされているモジュールを **表示** | 見る |
| **`modinfo`** | モジュールの **情報を表示** | 調べる |
| **`modprobe`** | モジュールを **ロード**（依存関係も解決） | 入れる |
| **`rmmod`** | モジュールを **アンロード**（取り外し） | 抜く |

> 💡 `insmod` というコマンドもあります（modprobeに似ているが依存関係を自分で解決しない）。試験では `modprobe` のほうがメインで出題されます。

#### 実行例

```bash
# 現状確認
$ lsmod | grep e1000
e1000               137216  0

# 詳細情報
$ modinfo e1000
filename:    /lib/modules/4.15.0/kernel/drivers/net/ethernet/intel/e1000/e1000.ko
description: Intel(R) PRO/1000 Network Driver
（以下省略）

# ロード（手動でドライバを追加）
# modprobe e1000

# アンロード
# rmmod e1000
```

#### 🎯 試験ポイント

| 問われ方 | 答え |
|---|---|
| ロード済みモジュールの確認 | **lsmod** |
| モジュールをロードする | **modprobe** |
| モジュールをアンロードする | **rmmod** |
| デバイスドライバの実体は? | **カーネルモジュール** |

## 1.2 Linuxの起動とシャットダウン

### 1.2.1 システムが起動するまでの流れ

#### ここで学ぶこと

**これが第1章の核心です。試験で必ず出ます。**

「電源ONからログイン画面が出るまで、内部で何が起きているか」を時系列で理解します。登場人物は5人。それぞれが「次の人にバトンを渡す」リレー方式です。

#### 起動シーケンス全体像

![Linux起動の物語 ─ 電源ONからログイン画面まで](images/12_boot_sequence.png)

#### 各登場人物の役割（詳細）

##### STEP 1: BIOS / UEFI

- **どこに居る**：マザーボード上のフラッシュROM
- **何をする**：
  1. POST（ハード自己診断）でメモリやCPUなどに問題ないか確認
  2. 起動ドライブ（HDD/SSD等）を探して、その先頭にある **ブートローダ** を読み込む
- **次に渡すバトン**：制御をブートローダに渡す

##### STEP 2: ブートローダ

- **どこに居る**：HDD/SSDの先頭部分
- **何をする**：
  1. HDD/SSDから **カーネル本体** を読み出す
  2. カーネルをメモリに展開して実行を開始する
- **代表的なソフト**：**GRUB 2**（Linux標準）
- **次に渡すバトン**：制御をカーネルに渡す

> 💡 **GRUB 2 は第2章で詳しく扱います**。今は「ブートローダの代表 = GRUB 2」とだけ覚えておけばOK。

##### STEP 3: カーネル

- **何をする**：
  1. メモリの初期化
  2. システムクロックの設定
  3. **initramfs**（初期RAMディスク）をマウントしてハードウェアを認識
  4. 最初のプロセスを起動（init または systemd）
- **以降の役割**：メモリに常駐し、システムの中心として動き続ける

##### STEP 3.5: initramfs（初期RAMディスク）

- **これは何?**：カーネルが起動直後に使う **一時的なルートファイルシステム**
- **何が入ってる?**：起動に必要な最小限のデバイスドライバとファイル群
- **なぜ必要?**：本物のHDD/SSDにアクセスするためのドライバを読み込む「橋渡し」が必要だから

> 💡 鶏と卵問題：HDDの中身を読むには HDD用ドライバが要る。でもそのドライバ自体がHDDの中にある。これを解決するのが initramfs。一時的にメモリ上に展開された小さなファイルシステムから起動を進める。

##### STEP 4: init / systemd

- **どこに居る**：メモリ上のプロセス（**PID 1**）
- **何をする**：
  1. 各種サービス（デーモン）を順次/並列に起動
  2. ログイン画面の準備
  3. 全プロセスの親玉として常駐

##### PID 1 とは

**PID（Process ID）** とは、プロセス1つひとつに振られる「背番号」のことです。`PID 1` は **最初に起動するプロセス** で、すべてのプロセスの親玉です。

- 旧来（SysVinit時代）：`/sbin/init` がPID 1
- 現代（systemd時代）：`systemd` プロセスがPID 1

#### 覚え方

> **「BBKii」**
> - **B**IOS/UEFI
> - **B**ootloader
> - **K**ernel
> - **i**nitramfs
> - **i**nit / systemd

#### 🎯 試験ポイント

試験ではほぼ確実に「**起動順を正しく並べよ**」が出ます。次の選択肢が正解：

```
✅ BIOS/UEFI → ブートローダ → カーネル → init/systemd
❌ init/systemd → BIOS/UEFI → ブートローダ → カーネル
❌ カーネル → init/systemd → ブートローダ → BIOS/UEFI
❌ BIOS/UEFI → カーネル → ブートローダ → init/systemd
```

### 1.2.2 起動時のイベント確認

#### ここで学ぶこと

「起動時にどんなことが起きたか」を後から確認する方法を学びます。`dmesg` と `journalctl` の2つのコマンドを使い分けます。

#### カーネルメッセージとは

Linuxカーネルは起動中、デバイス認識やエラー情報などを **メッセージとして出力** します。「USBメモリを認識しました」「ネットワークカードのリンクが上がりました」のような内容です。

このメッセージは、トラブルシューティング（「あれ?USBが認識されない?」など）に欠かせない情報です。

![起動時のメッセージを見る ─ dmesg と journalctl](images/13_dmesg_journalctl.png)

#### dmesg コマンド（古いシステム向け）

```bash
$ dmesg | head -20
[    0.000000] Linux version 4.15.0-42-generic ...
[    0.000000] Command line: BOOT_IMAGE=/boot/vmlinuz-4.15.0-42-generic ...
[    0.000000] KERNEL supported cpus:
[    0.000000]   Intel GenuineIntel
（以下省略）
```

- カーネルが出力したメッセージを **一時的に蓄えるバッファ**（リングバッファ）の内容を表示
- 起動後もメッセージが追加されていくので、**古いものは消えていく**
- **長時間稼働しているシステム**では、起動時のメッセージが既に消えている可能性あり

#### ログファイルにも残る

起動時のメッセージは、以下のファイルにも保存されています：

- `/var/log/messages`
- `/var/log/dmesg`
- `/var/log/boot.log`

これらはテキストなので `cat` で読めます（root権限が必要なことがあります）。

#### journalctl コマンド（systemd環境）

systemdを採用したシステムでは、**`journalctl`** コマンドが使えます。dmesgより便利で、メッセージが永続化されるので過去のログも見られます。

| コマンド | 動作 |
|---|---|
| `journalctl -k` | **dmesgと同様** にカーネルのバッファを表示 |
| `journalctl -kb` | **システム起動時** のカーネルメッセージを表示 ← 頻出 |
| `journalctl -b` | 現在の起動以降のすべてのログ |

#### 🎯 試験ポイント

| 問われ方 | 答え |
|---|---|
| 起動時のカーネルメッセージを見る（古い系） | **dmesg** |
| 起動時のカーネルメッセージを見る（systemd系） | **journalctl -kb** |
| 起動ログが保存されるファイル | `/var/log/messages`, `/var/log/dmesg`, `/var/log/boot.log` |

### 1.2.3 シャットダウンと再起動

#### ここで学ぶこと

システムを安全に停止・再起動するコマンドを学びます。試験では `shutdown` コマンドの **オプションの組み合わせ** がほぼ確実に出題されます。

#### なぜshutdownコマンドを使うのか

PCの電源ボタンを押せば電源は切れますが、それでは **動作中のプログラムが強制終了** されてしまいます。ファイル書込中だったらデータが壊れますし、データベースが起動中なら整合性が崩れます。

`shutdown` コマンドは：

1. ログインユーザーに **通知** する
2. 動作中のサービスを **順番に終了** する
3. ファイルシステムを **きれいに片付ける**
4. その後で安全に電源OFF

という手順で停止してくれます。

#### shutdown コマンドの構文

![shutdown コマンド ─ オプションと時間指定](images/14_shutdown_options.png)

```
shutdown [オプション] 時間 [メッセージ]
```

#### 主なオプション

| オプション | 説明 |
|---|---|
| **`-h`** | **シャットダウン** する（halt） |
| **`-r`** | シャットダウン後に **再起動** する（reboot） |
| `-f` | 次回起動時に fsck をスキップ |
| `-F` | 次回起動時に fsck を必ず実行 |
| **`-k`** | 実際には停止せず **警告メッセージのみ** 通知 |
| **`-c`** | 実行中のシャットダウンを **キャンセル** |

#### 時間の指定方法

- `now` → 即時実行
- `+N` → N分後（例: `+5` で5分後）
- `hh:mm` → 指定時刻（例: `22:00` で22時に）

#### 実行例

5分後にシャットダウン、警告メッセージ付き：
```bash
# shutdown -h +5 "Please logout immediately."
```

22時にシャットダウン：
```bash
# shutdown -h 22:00
```

即時再起動：
```bash
# shutdown -r now
```

警告のみ（実際には停止しない）：
```bash
# shutdown -k now 'Please logout immediately.'
```

実行中のシャットダウンをキャンセル：
```bash
# shutdown -c
```

#### 他のシャットダウン関連コマンド

| コマンド | 動作 |
|---|---|
| `halt` | システム停止 |
| `poweroff` | 電源OFF |
| `reboot` | 再起動 |
| `init 0` | システム停止（ランレベル0へ） |
| `init 6` | 再起動（ランレベル6へ） |
| `systemctl poweroff` | systemd環境での停止 |
| `systemctl reboot` | systemd環境での再起動 |

#### shutdown vs init/telinit

- `shutdown` → **ユーザに通知される**（推奨）
- `init` / `telinit` で停止・再起動 → **通知されない**（事前準備が必要）

#### 🎯 試験ポイント

試験ではコマンドを「書き下し」で問われるパターンが頻出：

- 「1時間後に再起動」→ `shutdown -r +60`
- 「即時シャットダウン」→ `shutdown -h now`
- 「警告だけ」→ `shutdown -k now "メッセージ"`
- 「実行中のキャンセル」→ `shutdown -c`

## 1.3 SysVinit

### ここで学ぶこと（1.3 全体）

**SysVinit** は、systemdが普及する前に長く使われていた **古い起動の仕組み** です。「今はもう使わないなら勉強しなくていいんじゃ?」と思うかもしれませんが、**LPICでは両方の知識が問われます** し、古いシステムを扱うこともあるので押さえる必要があります。

このセクションでは、SysVinitの起動の流れ、ランレベル、起動スクリプトの3つを順に見ていきます。

### 1.3.1 SysVinitによる起動

#### SysVinitとは

**SysVinit（System Five Init）**
UNIXのSystem V（システム5）で採用されていた起動の仕組みに基づく、伝統的な起動システムです。

- 長らくLinuxの標準的な起動方式だった
- 現在は **systemd** に置き換わってきている

#### 起動の流れ

![SysVinit の起動フロー](images/15_sysvinit_flow.png)

1. **init プロセスが起動（PID 1）** → カーネルが最初に起動するプロセス
2. **`/etc/inittab` を読み込む** → デフォルトのランレベルなどの設定
3. **`/etc/rc.sysinit` を実行** → システム共通の初期化スクリプト
4. **`/etc/rc<N>.d/` のスクリプトを順次実行** → ランレベルN用のサービス群を起動

#### /etc/inittab の例

```
# Default runlevel.
id:3:initdefault:

# System initialization.
si::sysinit:/etc/rc.d/rc.sysinit

l0:0:wait:/etc/rc.d/rc 0
l1:1:wait:/etc/rc.d/rc 1
l2:2:wait:/etc/rc.d/rc 2
l3:3:wait:/etc/rc.d/rc 3
（以下省略）

# Run gettys in standard runlevels
1:2345:respawn:/sbin/mingetty tty1
```

`id:3:initdefault:` → **デフォルトのランレベルを3に設定** という意味です。

> 💡 SysVinitを使っていない最近のシステムでは、`/etc/inittab` は存在しないか、中身が空っぽです。

#### SysVinitの欠点（systemdが生まれた理由）

- サービスは **順番に直列で起動** していく
- 1つのサービスの起動が遅れると、後続が全部待たされる
- 結果として **起動完了に時間がかかる**

これを改善するために `systemd` や `Upstart` が登場しました。

> 💡 **Upstart** = SysVinitとsystemdの中間世代。Red Hat、Ubuntuで一時期使われたが、現在はsystemdに移行済み。

---

### 1.3.2 ランレベル

#### ランレベルとは

SysVinit環境にある、**システムの動作モード** のことです。0〜6の数字（と S/s）で表され、ランレベルごとに「どのサービスを起動するか」が決まっています。

たとえば：
- ランレベル3 → サーバ用途。各種ネットワークサーバが起動、テキスト画面
- ランレベル5 → クライアント用途。GUIのログイン画面が起動

#### ランレベル早見表（Red Hat系）

![ランレベル早見表](images/16_runlevels.png)

| ランレベル | 説明 |
|---|---|
| **0** | **停止**（シャットダウン） |
| **1** | **シングルユーザーモード**（メンテナンス用） |
| 2 | マルチユーザーモード（NFSサーバなし） |
| **3** | **マルチユーザーモード（テキストログイン）** ← サーバ用標準 |
| 4 | 未使用 |
| **5** | **マルチユーザーモード（グラフィカルログイン）** |
| **6** | **再起動** |
| **S または s** | シングルユーザーモード（別名） |

#### Ubuntu / Debian系のランレベル

Ubuntu/Debian系では、ランレベル 2〜5 はすべて同じ意味（マルチユーザーモード）として扱われます。試験では **Red Hat系のランレベル** が基本で問われます。

#### シングルユーザーモード

- **rootユーザーだけ** が利用できる特殊な状態
- 一般ユーザーはログインできない
- **システムメンテナンス** や緊急対応時に使う
- パスワード忘れの復旧などにも利用される

#### 覚え方

> **「両端と3・5」**
> - **0 = 停止、6 = 再起動**（両端）
> - **1 = シングルユーザー**（メンテナンス）
> - **3 = テキストログイン**（サーバ用）
> - **5 = グラフィカルログイン**（クライアント用）
> - 4は無視でOK

#### 🎯 試験ポイント

「ランレベル○は何?」「○○するにはどのランレベルか」のパターンで頻出。**0, 1, 3, 5, 6** は確実に覚えること。

---

### 1.3.3 ランレベルの確認と変更

#### 現在のランレベルを確認

```bash
# runlevel
N 3
```

出力は2つの値：
- 左：**前のランレベル**（`N` = なし、つまり起動後変更していない）
- 右：**現在のランレベル**（この例では3）

別の方法：

```bash
$ who -r
         run-level 3  2018-11-25 12:34
```

#### ランレベルを変更

`init` または `telinit` コマンドを使います（**root権限が必要**）。

```bash
# init 1      # シングルユーザーモードへ
# telinit 5   # グラフィカルログインへ
# init s      # シングルユーザーモードへ（別記法）
```

#### ユーザへの通知（wallコマンド）

ランレベル変更前に、ログイン中のユーザに通知すべきです。
`wall` コマンドで全ユーザーの端末にメッセージを送れます：

```bash
# wall 'This system is going down to maintenance mode! Please logout!'
```

ユーザーの端末画面：

```
Broadcast message from root (tty1) (Sun Nov 25 18:41:09 2018):

This system is going down to maintenance mode! Please logout!
```

#### 🎯 試験ポイント

- 「現在のランレベル確認は?」 → `runlevel` または `who -r`
- 「ランレベルを1に変更する」 → `init 1` または `telinit 1` または `init s`
- 「全ユーザに通知するコマンド」 → `wall`

---

### 1.3.4 起動スクリプトによるサービスの管理

#### スクリプトの場所

SysVinitでは、サービスの起動は2つのディレクトリの組み合わせで管理されます：

![SysVinit のサービス起動スクリプトの構造](images/17_initd_rcd.png)

- **`/etc/init.d/`** ← 起動スクリプトの **本体**（実体ファイル）
- **`/etc/rc<ランレベル>.d/`** ← 各ランレベルで実行するスクリプトの **シンボリックリンク**

#### /etc/rc3.d/ の中身

ランレベル3で起動するサービスのリンクが並んでいます：

```bash
$ ls /etc/rc3.d
K01dnsmasq         K25squid           K87multipathd     S10network
K01smartd          K30spamassassin    K87named          S12syslog
K02NetworkManager  K35dovecot         K87portmap        S13cpuspeed
S26acpid           S26haldaemon       S90crond
（以下省略）
```

#### ファイル名のルール（試験頻出）

| プレフィックス | 意味 |
|---|---|
| **`S`** | **Start**（起動する） |
| **`K`** | **Kill**（停止する） |

数字 = **実行順序**

例：`S10network` = ランレベル変更時、**10番目に起動** されるネットワークサービス
例：`K01dnsmasq` = ランレベル変更時、**1番目に停止** されるDNSサービス

#### スクリプトの実体（シンボリックリンク）

```bash
$ ls -l /etc/rc3.d/S90crond
lrwxrwxrwx 1 root root 15 Jun 19 23:20 /etc/rc3.d/S90crond -> ../init.d/crond
```

→ `/etc/init.d/crond` を指すシンボリックリンクになっています。

#### サービスの起動・停止

```bash
# /etc/init.d/httpd start     # httpdサービスを起動
Starting httpd:    [  OK  ]

# /etc/init.d/httpd stop      # httpdサービスを停止
Stopping httpd:    [  OK  ]

# /etc/init.d/httpd restart   # 再起動
# /etc/init.d/httpd status    # 状態確認
```

#### 🎯 試験ポイント

| 問われ方 | 答え |
|---|---|
| 起動スクリプトの本体場所 | **/etc/init.d/** |
| ランレベル別スクリプトの場所 | **/etc/rc&lt;N&gt;.d/** |
| `S` で始まるファイル | **Start**（起動） |
| `K` で始まるファイル | **Kill**（停止） |
| httpdサービスを起動するコマンド | `/etc/init.d/httpd start` |

---

### 1.3.5 デフォルトのランレベル設定

デフォルトのランレベルは **`/etc/inittab`** に記述されています。

```
# 0 - halt
# 1 - Single user mode
# 2 - Multiuser, without NFS
# 3 - Full multiuser mode
# 4 - unused
# 5 - X11
# 6 - reboot
#
id:3:initdefault:           ← この行で指定
```

デフォルトを `5`（グラフィカル）に変えるなら：

```
id:5:initdefault:
```

⚠ **`0` や `6` を指定すると、システムは正常に起動しません**（停止または再起動を繰り返します）。

設定の即時反映：

```bash
# init q
```

#### 🎯 試験ポイント

- 「デフォルトランレベルの設定ファイル」 → **/etc/inittab**
- 「id:N:initdefault: のNはランレベル」

## 1.4 systemd

### 1.4.1 systemdの概要

#### systemdとは

SysVinitに代わる、**現在主流の** 起動・サービス管理の仕組みです。最初のプロセス（PID 1）は `init` ではなく `systemd` プロセスになります。

#### systemdの利点 ─ なぜ生まれたか

SysVinitの欠点（順次起動で遅い）を解決するために設計されました：

![SysVinit と systemd ─ 起動の仕方が根本的に違う](images/18_sysvinit_vs_systemd.png)

| 項目 | SysVinit | systemd |
|---|---|---|
| サービス起動 | **順次**（直列） | **並列** |
| 依存関係 | 管理者が手動管理 | 自動処理 |
| 不要なサービス | 設定で起動 | 必要なサービスのみ起動 |
| 起動時間 | 長い | **短い** |

#### systemd 関連の主なデーモンプロセス

systemdは1つのプロセスではなく、複数のデーモンプロセスが連携して動いています：

| プロセス | 説明 |
|---|---|
| `systemd` | systemdのメインプロセス（PID 1） |
| `systemd-journald` | ジャーナル（ログ）管理 |
| `systemd-logind` | ログイン処理 |
| `systemd-networkd` | ネットワーク管理 |
| `systemd-timesyncd` | システムクロック同期 |
| `systemd-resolved` | 名前解決（DNS） |
| `systemd-udevd` | デバイス動的検知（前述のudev） |

> 💡 **デーモン（daemon）**：メモリ上に常駐するプロセス。サーバサービスを提供する常駐型プログラム。名前の末尾に「d」が付くことが多い（sshd, crond, httpd, acpid, udevd...）。

#### Unit（ユニット）とは

systemdの最大の特徴は、**すべての処理を「Unit」という単位で管理する** ことです。Unitには種類があります：

![systemd の Unit と target](images/19_systemd_units.png)

| Unit種類 | 説明 |
|---|---|
| **`.service`** | 各種サービスを起動する（例：`httpd.service`） |
| `.device` | 各種デバイスを表す |
| `.mount` | ファイルシステムをマウントする |
| `.swap` | スワップ領域を有効にする |
| **`.target`** | **複数のUnitをグループ化する**（旧ランレベル相当） |
| `.timer` | 指定した日時や間隔で処理を実行する（cron代替） |

#### 🎯 試験ポイント

- 「systemdの最初のプロセスPIDは?」 → **1**
- 「systemdで起動を管理する単位は?」 → **Unit**
- 「旧ランレベル相当のUnit種別は?」 → **target**
- 「常駐型プロセスを何と呼ぶ?」 → **デーモン**

---

### 1.4.2 systemdの起動手順

#### default.target ─ 起動の起点

システム起動時、systemdはまず **`default.target`** という Unit を処理します。
このファイルは `/etc/systemd/system/` 以下にあり、実体は **シンボリックリンク** です：

```bash
$ ls -l /etc/systemd/system/default.target
lrwxrwxrwx. 1 root root 36 2月 8 04:51 /etc/systemd/system/default.target 
  -> /lib/systemd/system/graphical.target
```

→ `default.target` は `graphical.target`（グラフィカルログイン）を指している、と読みます。

#### ターゲット（target）とランレベルの対応

| ランレベル | ターゲット |
|---|---|
| 0 | `poweroff.target` |
| 1 | `rescue.target` |
| 2, 3, 4 | `multi-user.target` |
| 5 | `graphical.target` |
| 6 | `reboot.target` |

#### ターゲットの階層構造

target は **入れ子構造** で依存関係を持っています：

```
graphical.target
  └── multi-user.target
        └── basic.target
              └── (ランレベル問わず必要なサービス)
```

`graphical.target` は `multi-user.target` を含み、`multi-user.target` は `basic.target` を含む。階層的に依存関係が組まれているので、上位のターゲットを起動すれば、必要な下位のターゲットも自動的に起動します。

#### Unit ファイルの場所（重要）

| 場所 | 用途 |
|---|---|
| **`/lib/systemd/system/`** | **オリジナル** のUnit定義ファイル（パッケージが提供） |
| **`/etc/systemd/system/`** | **ホストごとの修正・カスタマイズ** したUnit定義ファイル |

> 💡 試験で問われる典型：「ホストごとに設定を変更する場合はどこにファイルを置く?」 → **/etc/systemd/system/**
>
> パッケージインストールで自動的に置かれるオリジナルが `/lib/systemd/system/` に。それを上書きしたい個別カスタマイズが `/etc/systemd/system/` に。

#### デフォルトターゲットの変更

例：グラフィカルログインを既定にする：

```bash
# ln -s /lib/systemd/system/graphical.target /etc/systemd/system/default.target
```

`ln -s` でシンボリックリンクを張り直すだけ。

#### 🎯 試験ポイント

| 問われ方 | 答え |
|---|---|
| 起動時に最初に処理されるUnit | **default.target** |
| グラフィカル相当のターゲット | **graphical.target** |
| マルチユーザー相当のターゲット | **multi-user.target** |
| 個別カスタマイズの置き場所 | **/etc/systemd/system/** |
| パッケージ提供のオリジナル置き場所 | **/lib/systemd/system/** |

---

### 1.4.3 systemctlによるサービス管理

#### systemctlコマンド

systemdでサービスを管理する標準コマンドです。これ一つで起動、停止、再起動、自動起動設定、状態確認、すべてできます。

![systemctl ─ systemd のサービス管理コマンド](images/20_systemctl_commands.png)

```
systemctl サブコマンド [Unit名] [-t 種類]
```

#### 主なサブコマンド（試験頻出 ─ 動詞3 + 永続化2 + 確認1）

##### 動詞3 ─ サービスを操作する

| サブコマンド | 説明 |
|---|---|
| **`start`** | サービスを **起動** |
| **`stop`** | サービスを **停止** |
| **`restart`** | サービスを **再起動** |

##### 永続化2 ─ 起動時の自動起動設定

| サブコマンド | 説明 |
|---|---|
| **`enable`** | **システム起動時に自動起動する** |
| **`disable`** | システム起動時に自動起動しない |

##### 確認1 ─ 状態を見る

| サブコマンド | 説明 |
|---|---|
| **`status`** | サービスの稼働状況を表示（active / inactive など） |

##### その他よく使うサブコマンド

| サブコマンド | 説明 |
|---|---|
| `reload` | サービスの設定を再読み込み |
| `is-active` | active か inactive かだけを表示 |
| `mask` | Unit をマスク（手動でも起動できなくする） |
| `unmask` | マスクを解除 |
| `list-units` | 起動中のすべての Unit と状態を表示 |
| `list-unit-files` | すべての Unit を表示 |
| `list-dependencies` | Unitの依存関係を表示 |
| `reboot` | システム再起動 |
| `poweroff` | システム停止 |

#### 実行例

```bash
# Postfixサービスを起動
# systemctl start postfix.service

# 状態確認
# systemctl status postfix.service
● postfix.service - Postfix Mail Transport Agent
   Loaded: loaded (/lib/systemd/system/postfix.service; enabled; vendor preset: enabled)
   Active: active (exited) since Tue 2018-11-13 08:59:19 UTC; 2min 14s ago
  Process: 1672 ExecStart=/bin/true (code=exited, status=0/SUCCESS)
 Main PID: 1672 (code=exited, status=0/SUCCESS)

# 起動中か簡易確認
$ systemctl is-active postfix.service
active

# システム起動時に自動起動するよう設定
# systemctl enable postfix.service
ln -s '/usr/lib/systemd/system/postfix.service' 
      '/etc/systemd/system/multi-user.target.wants/postfix.service'

# 自動起動しないよう設定
# systemctl disable postfix.service
```

> 💡 `.service` は省略可能です。`systemctl start postfix` でも同じ意味になります。

#### サービス一覧の表示

```bash
# すべてのサービスを表示
$ sudo systemctl list-unit-files -t service
UNIT FILE                                           STATE
accounts-daemon.service                            enabled
acpid.service                                       disabled
alsa-restore.service                               static
alsa-state.service                                 static
alsa-utils.service                                 masked

# 自動起動するサービスだけ
$ sudo systemctl list-unit-files -t service --state=enabled
```

#### サービスの状態

| 状態 | 意味 |
|---|---|
| `enabled` | 自動起動する |
| `disabled` | 自動起動しない |
| `static` | 他のUnitから呼ばれて起動する（自動起動設定不可） |
| `masked` | マスク状態（手動でも起動できない） |

#### 🎯 試験ポイント

「httpdサービスを起動するコマンド」「自動起動を有効にするコマンド」「状態確認するコマンド」のパターンがほぼ確実に出ます。

| 問われ方 | 答え |
|---|---|
| httpdを起動する | `systemctl start httpd` または `systemctl start httpd.service` |
| 自動起動を有効に | `systemctl enable httpd` |
| 自動起動を無効に | `systemctl disable httpd` |
| 状態を確認 | `systemctl status httpd` |
| サービス一覧を表示 | `systemctl list-unit-files -t service` |

## 📝 全体まとめ ─ ここまでの学習内容

このセクションを終えた時点で、次のことができるようになっているはずです：

1. **PCのハードウェア構成** を5つ以上挙げて説明できる
2. **BIOS/UEFI** の3つの役割を言える
3. **`/proc/`, `/dev/`** の意味と例を答えられる
4. **`lspci`, `lsusb`, `lsmod`, `modprobe`** などのコマンドの用途を区別できる
5. **udev** の役割を一言で言える
6. **起動シーケンス**（BIOS→ブートローダ→カーネル→init/systemd）を正しい順で言える
7. **dmesg と journalctl** の使い分けができる
8. **shutdownコマンド** のオプション（-h, -r, -k, -c）を区別できる
9. **ランレベル** 0/1/3/5/6 を答えられる
10. **/etc/inittab, /etc/init.d/, /etc/rc&lt;N&gt;.d/** の役割を区別できる
11. **systemd の Unit と target** の概念を説明できる
12. **systemctl の主要サブコマンド**（start/stop/restart/enable/disable/status）を使える

これらができていれば、研修当日のテストで困ることはほぼないはずです。
