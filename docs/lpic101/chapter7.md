## 7.1 シェル環境のカスタマイズ

### ここで学ぶこと

- ユーザーの作業環境を決める **変数**(環境変数とシェル変数)の違い
- シェルの動作を切り替える **オプション**(set)
- コマンドに別名を付ける **エイリアス** と、処理をまとめる **関数**
- 設定を自動化する **bashの設定ファイル** と、その **実行順序**

「シェル」とは、第3章で学んだ通り、ユーザーが入力したコマンドを受け取ってOSに渡す **窓口** のプログラムです(Linuxでは **bash** が標準)。

たとえるなら、シェルは「店の受付係」のような存在です。あなた(ユーザー)が「これをやって」とお願いすると、受付係(シェル)が奥のスタッフ(OS)に取り次いでくれます。ところが、同じ受付係でも、人によって「日本語で応対してほしい」「呼ばれ方を変えてほしい」「よく頼む注文はひとことで通したい」など、好みは様々です。この節では、その **受付係(シェル)を自分好みに教育する** 方法 ── つまりシェル環境のカスタマイズを学びます。

カスタマイズの道具は大きく4つあります。「環境を覚えさせる入れ物」である **変数**、「動き方のスイッチ」である **オプション**、「よく使う命令のあだ名」である **エイリアス**、「一連の作業に名前を付ける」**関数** です。そして、これらを毎回手で設定しなくて済むように **設定ファイル** が用意されています。順に見ていきましょう。

### 7.1.1 環境変数とシェル変数

#### 変数は2種類 ─ 値が「どこまで」届くか

シェルの環境は、たくさんの **変数** によって決まります。変数とは、値を一時的に入れておく **名前付きの入れ物** のことです(第3章でも登場しました)。たとえば「PATH」という名前の入れ物には「コマンドを探しに行く場所のリスト」が、「HOME」という入れ物には「自分のホームディレクトリの場所」が入っています。

この変数には **環境変数** と **シェル変数** の2種類があり、違いは **その値がどこまで届くか(有効範囲)** です。ここでいう「プロセス」とは、実行中のプログラム1つ1つのこと、「子プロセス」とは、あるシェルから起動された別のプログラムや別のシェルのことです(第4章のプロセスの話を思い出してください)。

![環境変数とシェル変数](images/76_env_vs_shell_var.png)

- **環境変数**: そのシェル自身と、**そこから起動されたすべてのプロセス**(コマンドや別のシェル)で有効。いわば「家じゅうの全員に伝わる連絡事項」です。代表例は、コマンドを検索するディレクトリのリストを格納する **PATH** や、ホームディレクトリのフルパスを表す **HOME**
- **シェル変数**: **そのシェルの中だけ** で有効。「自分の部屋でだけ通じるメモ」のようなもので、他のシェルや、そこから起動したコマンドには引き継がれません

身近な例で言えば、シェル変数は「自分の手元のメモ帳」、環境変数は「家族の伝言板」です。手元のメモは自分しか見られませんが、伝言板に書けば、後から起動した子プロセス(=家族)にも内容が伝わります。

#### export ─ シェル変数を環境変数に昇格させる

最初に作った変数は、すべてシェル変数です(手元のメモの状態)。これを環境変数にする(伝言板に書き写す)操作が **export** です。シェル変数を export(エクスポート)すると環境変数になり、そのシェルから起動したコマンドや別のシェルでも使えるようになります。

```bash
$ VAR=hello       # まずシェル変数として定義(手元のメモ)
$ export VAR      # 環境変数に昇格(伝言板に転記 → 子プロセスでも有効になる)
```

> 💡 **覚え方Hack ─ export は「輸出」**
> `export` は英語で「輸出」。手元(シェル内)だけで使っていた変数を、**輸出して初めて外(子プロセス)に届く** ようになる、とイメージすると、環境変数とシェル変数の違いが一発で整理できます。

#### 変数を確認・操作するコマンド

設定されている変数を確認したり、操作したりするコマンドを整理します。「いま手元のメモと伝言板に何が書いてあるか」を見たいときに使います。

![変数を扱うコマンド](images/77_var_commands.png)

| コマンド | 何をする |
|---|---|
| **printenv** / **env** | **環境変数**(伝言板)を表示する(`printenv 変数名` でその値だけ表示) |
| **set** | **シェル変数 + 環境変数**(さらに関数も)を表示する。つまり手元のメモも伝言板も全部 |
| **export** | シェル変数を環境変数にする(子プロセスへ引き継ぐ) |
| **unset** | 変数や関数を削除する(メモを消す) |

> 💡 「環境変数だけ見たい → `env` / `printenv`」「全部(シェル変数も)見たい → `set`」と覚えましょう。`printenv HOME` のように変数名を付けると、その値だけが表示されます。`env` と `printenv` はほぼ同じ働きをすると考えて差し支えありません。

#### 📌 試験ポイント

| 問われ方 | 答え |
|---|---|
| シェルとその子プロセスで有効な変数は? | **環境変数** |
| そのシェル内だけで有効な変数は? | **シェル変数** |
| シェル変数を環境変数にするコマンドは? | **export** |
| 環境変数を表示するコマンドは? | **printenv / env** |
| シェル変数も含めて表示するコマンドは? | **set** |
| 変数を削除するコマンドは? | **unset** |
| コマンド検索パスを格納する環境変数は? | **PATH** |
| ホームディレクトリを表す環境変数は? | **HOME** |

### 7.1.2 シェルのオプション

#### set -o / +o ─ 動作のスイッチ

シェルには、動作のクセを切り替える **オプション** がいくつもあります。たとえば「ファイル名の `*` を自動展開するかどうか」「うっかりCtrl+Dを押してもログアウトしないか」といった、シェルの振る舞いのオン/オフです。家電のリモコンにある「静音モード」「省エネモード」のようなスイッチ群だと思ってください。これらを切り替えるのが **set** コマンドです。

![シェルのオプション](images/78_set_options.png)

```
書式: set [-o][+o] [オプション]
```

ここが最大の引っかけポイントです。**`-o`(マイナス)で有効、`+o`(プラス)で無効** になります。プラスは普通「追加=オン」を連想させますが、シェルのオプションでは逆。プラスで機能を「取り消す(オフ)」のです。直感と逆なので、試験でも実務でも要注意です。

| オプション | 説明 |
|---|---|
| **allexport** | 作成・変更した変数を自動的にエクスポートする(全部を自動で伝言板へ) |
| **noclobber** | リダイレクト(`>`)による上書きを禁止する(既存ファイルを誤って消さない) |
| **noglob** | メタキャラクタ(`*` `?`)によるファイル名展開を無効にする |
| **ignoreeof** | Ctrl+D でログアウトしないようにする |
| **emacs** | emacs風のキーバインドにする |
| **vi** | vi風のキーバインドにする |

ここで「メタキャラクタ」とは、第3章で学んだ `*` や `?` のような **特別な意味を持つ記号** のことです。通常 `*.sh` と書くと、シェルが「.shで終わる全ファイル」に展開してくれますが、`noglob` を有効にするとこの展開が止まり、`*.sh` という文字どおりの名前として扱われます。

```bash
$ set -o noglob    # ファイル名展開を無効にする
$ set +o noglob    # 元に戻す(有効にする)
$ set -o           # 現在のオプション設定を一覧表示
```

> 💡 **覚え方Hack ─ マイナスで点ける、プラスで消す**
> 電気のスイッチと逆だと思うと混乱します。「`-o` = optionを on」「`+o` = onを取り消す(off)」と、`o` を on の頭文字として結びつけると覚えやすいです。

#### 📌 試験ポイント

| 問われ方 | 答え |
|---|---|
| シェルのオプションを切り替えるコマンドは? | **set** |
| オプションを有効にするのは? | **-o**(マイナス) |
| オプションを無効にするのは? | **+o**(プラス) |
| メタキャラクタ展開を無効にするオプションは? | **noglob** |
| リダイレクトの上書きを禁止するオプションは? | **noclobber** |
| Ctrl+Dでログアウトしないようにするのは? | **ignoreeof** |
| 現在のオプションを一覧表示するには? | **set -o** |

### 7.1.3 エイリアス

#### alias ─ コマンドに別名を付ける

毎回 `ls -l` と打つのは面倒です。そこで、よく使うコマンドに **あだ名(別名)** を付けておけば、短い名前で呼び出せます。これが **エイリアス** です。「いつものやつ」と言えば常連客に通じる、あの感覚をコマンドで実現する仕組みだと思ってください。設定は **alias** コマンドで行います。

![エイリアス](images/79_alias.png)

```bash
$ alias ls='ls -l'             # 以後 ls と打つと ls -l が実行される
$ alias lsless='ls -l | less'  # 複数のコマンドをまとめて1つの名前にもできる
```

ここで値を **シングルクォート(`'`)** で囲んでいるのには理由があります。囲まないと、**コマンドとオプションの間のスペースをシェルが区切りと解釈** してしまい、意図通りに登録されません。「ひとつのまとまりだよ」と教えるために引用符で囲む、と理解しておきましょう(第3章のクォートの話と同じ考え方です)。

#### エイリアスの解除と一時的な無効化

| やりたいこと | 方法 |
|---|---|
| エイリアスを解除する | `unalias エイリアス名` |
| すべてのエイリアスを解除する | `unalias -a` |
| そのときだけ無効にする(解除はしない) | コマンドの前に **`\`** を付ける |

ここで「解除」と「一時的な無効化」は別物です。`unalias` は登録そのものを消してしまう(あだ名を廃止する)操作。一方、「今回だけ元のコマンドを使いたい(あだ名ではなく本名で呼びたい)」ときは、コマンドの直前に **`\`** を付けます。

```bash
$ unalias lsless     # lsless というあだ名を廃止
$ \ls                # ls のエイリアスを今回だけ無視して、素の ls を実行
```

> ⚠ **超頻出 ─ 一時的な無効化は `\`**
> 「エイリアスを解除せずに、今回だけ元のコマンドを使いたい」ときは、コマンドの直前に **`\`** を付けます。試験では似た選択肢で揺さぶってきます。`unalias` だと完全に解除されてしまい、`#` を付けるとコメント行とみなされて実行されず、`'` や `` ` `` で囲むのも誤りです。「今回だけ素のコマンド → バックスラッシュ」とセットで覚えましょう。

#### 📌 試験ポイント

| 問われ方 | 答え |
|---|---|
| コマンドに別名を付けるコマンドは? | **alias** |
| エイリアスを解除するコマンドは? | **unalias** |
| すべてのエイリアスを解除するには? | **unalias -a** |
| エイリアスを一時的に無効にするには? | コマンド前に **`\`** |
| alias定義で値を `'` で囲む理由は? | **スペースをシェルに解釈させないため** |

### 7.1.4 関数の定義

#### function ─ コマンドの組み合わせに名前を付ける

エイリアスは「コマンドのあだ名」でしたが、もっと込み入った処理 ── 複数のコマンドを組み合わせたり、引数を受け取って動きを変えたり ── をまとめたいときは **関数** を使います。「いくつかの手順をまとめて1つの作業名にする」もので、料理でいえば「下ごしらえ」という一語に複数の手順を込めるようなイメージです。bashの組み込みコマンド **function** で定義します。

![関数の定義](images/80_function.png)

```
書式: [function] 関数名() { コマンド; }
```

`function` という単語は省略できます。ひとつ注意したいのが、**`{` の後ろと `}` の前にはスペースが必要** な点です。これを忘れるとエラーになります。

```bash
$ function lslink() { ls -l $1 | grep '^l'; }    # 引数($1)も使える
$ lslink /etc                                     # 関数名を打てば実行される
```

ここで `$1` は「関数に渡された1番目の引数」を表します(詳しくは7.2で扱います)。上の例なら `lslink /etc` と実行すると、`$1` が `/etc` に置き換わり、`/etc` の中のシンボリックリンクだけを一覧表示します。

#### 関数の表示・削除

| やりたいこと | 方法 |
|---|---|
| 定義されている関数を表示 | `declare -f`(`set` でも変数に続いて表示される) |
| 関数を削除 | `unset 関数名`(変数の削除と同じ) |

> ⚠ **bashは変数名と関数名を区別しません**。同じ名前を変数にも関数にも使うと衝突するので、名前が重複しないよう注意が必要です。また、関数が使えるのは **定義したシェルの中だけ**(エイリアスと同様、外には持ち出せない)です。削除が変数と同じ **`unset`** である点もよく問われます。

#### 📌 試験ポイント

| 問われ方 | 答え |
|---|---|
| 関数を定義するコマンドは? | **function**(省略可) |
| 関数定義で必要な記法上の注意は? | **`{` の後ろと `}` の前にスペース** |
| 関数内で1番目の引数を参照するには? | **$1** |
| 定義済みの関数だけを表示するには? | **declare -f** |
| 関数を削除するコマンドは? | **unset** |
| 関数が有効な範囲は? | **定義したシェル内のみ** |

### 7.1.5 bashの設定ファイル

#### 設定を自動化するファイルたち

ここまでで、変数・エイリアス・関数で環境をカスタマイズできることを学びました。しかし、シェルを起動するたびにこれらを手で打ち直すのは、毎朝出社のたびに机の配置をゼロから組み直すようなもので、非効率です。そこで、**起動時にそれらの設定を自動で実行してくれるファイル** が用意されています。これが設定ファイルです(中身自体がシェルスクリプトになっています)。

![bashの設定ファイル](images/81_bash_config_files.png)

ポイントは **`/etc` 以下=全ユーザー共通**、**`~`(ホーム)以下=ユーザー個別** という住み分けです。`/etc` は「会社全体の就業規則」、`~`(自分のホーム)は「自分専用の手帳」と考えるとわかりやすいでしょう。

**全ユーザー共通(/etc 以下)**

| ファイル | 説明 |
|---|---|
| **/etc/profile** | ログイン時に実行され、全ユーザーから参照される |
| **/etc/bash.bashrc** | bash起動時に実行され、全ユーザーから参照される |
| **/etc/bashrc** | `~/.bashrc` から参照される |

**ユーザー個別(~ 以下)**

| ファイル | 説明 |
|---|---|
| **~/.bash_profile** | ログイン時に実行される |
| **~/.bash_login** | `~/.bash_profile` が無い場合、ログイン時に実行される |
| **~/.profile** | 上の2つが無い場合、ログイン時に実行される |
| **~/.bashrc** | bash起動時(対話型)に実行される |
| **~/.bash_logout** | ログアウト時に実行される |

> 💡 `/etc` 以下を変更すると **全ユーザーに影響** が及びます(就業規則を書き換えると全社員に関わる)。個人の好みはホームディレクトリ以下のファイルで設定します。`~` はそのユーザーのホームディレクトリ(自分の部屋)を表す記号です。なお、ファイル名の先頭の `.`(ドット)は「隠しファイル」を意味し、通常の `ls` では表示されません(第3章の復習)。

#### 📌 試験ポイント

| 問われ方 | 答え |
|---|---|
| 全ユーザー共通の設定ファイルはどこ? | **/etc 以下** |
| ユーザー個別の設定ファイルはどこ? | **~(ホーム)以下** |
| ログイン時に全ユーザーが参照するのは? | **/etc/profile** |
| ログアウト時に実行されるのは? | **~/.bash_logout** |
| 対話型bash起動時に実行される個人ファイルは? | **~/.bashrc** |

### 7.1.6 bash起動時における設定ファイルの実行順序

#### ログインシェルと対話型シェルで違う

前項のファイルたちは、いつでも全部読まれるわけではありません。**どういう状況でシェルが起動したか** によって、読まれるファイルと順番が変わります。これが試験頻出のテーマです。

鍵になるのが **ログインシェル** と **対話型シェル** の区別です。

- **ログインシェル**: ログインしたときに最初に起動するシェル。「出社して建物に入る」瞬間に動くイメージ。`ps` コマンドで見ると **`-bash`** のように頭に `-` が付きます
- **対話型シェル**: すでにログインした状態で、`bash` と打ったり端末エミュレータ(画面上のターミナル)を開いたときに起動するシェル。「すでに社内にいて、別の部屋に移る」イメージ

![設定ファイルの実行順序](images/82_config_order.png)

**ログインシェルの場合**(ログイン時)

1. **/etc/profile** を読み込んで実行(あれば)── まず会社の就業規則
2. その中で **/etc/bash.bashrc** を読み込んで実行(あれば)
3. その後、**~/.bash_profile → ~/.bash_login → ~/.profile** の順に探し、**最初に見つかった1つだけ** を実行 ── 自分の手帳

ここで大事なのは、個人用の3ファイルは **上から順に探して、最初に見つかったものだけ** が読まれる点です。3つ全部が順に実行されるわけではありません(`~/.bash_profile` があれば、残り2つは読まれない)。

**対話型シェルの場合**(`bash` コマンドや端末起動時)

- **~/.bashrc** があれば、これを読み込んで実行

つまり「ログイン時は profile 系」「すでにログイン済みで開いた端末は ~/.bashrc」という住み分けです。

> 💡 **覚え方Hack ─ profile はログイン、bashrc は起動ごと**
> `profile` は「ログインのときの身支度(その日1回)」、`bashrc` は「bashを起動するたびの設定(端末を開くたび)」とイメージ。`rc` は run command(実行するコマンド)の略で、UNIX系の設定ファイルによく付く接尾辞です。

#### 📌 試験ポイント

| 問われ方 | 答え |
|---|---|
| ログインシェルが最初に読むのは? | **/etc/profile** |
| その後に個人設定として探す順は? | **~/.bash_profile → ~/.bash_login → ~/.profile** |
| 上の3つはいくつ実行される? | **最初に見つかった1つだけ** |
| 対話型シェルが読む個人ファイルは? | **~/.bashrc** |
| ログインシェルは `ps` でどう見える? | 頭に **`-`** が付く(`-bash`) |

#### 📝 ここまでのまとめ

- 変数は **環境変数(子プロセスにも有効)** と **シェル変数(そのシェル内のみ)**。**export** で昇格(手元のメモ→伝言板)
- 確認: **printenv / env**(環境変数)、**set**(シェル変数も)。削除は **unset**
- **set -o** で有効・**+o** で無効(noglob, noclobber, ignoreeof など)。プラスで無効が引っかけ
- **alias** で別名、**unalias** で解除、**`\`** で一時的に無効化
- **function** で関数定義(`{ }` の内側にスペース、引数は `$1`、削除は **unset**、定義シェル内のみ有効)
- 設定ファイル: **/etc 以下=全員 / ~ 以下=個人**。順序は **ログインシェル(profile系) vs 対話型(~/.bashrc)**

## 7.2 シェルスクリプト

### ここで学ぶこと

- **シェルスクリプト** の作り方と、4つの **実行方法**
- スクリプトに値を渡す **引数(位置パラメータ)** と、成否を表す **戻り値 $?**
- ファイルや数値を判定する **test コマンド**
- 処理を分岐・繰り返す **制御構造**(if / case / for / while / read)
- スクリプトの **実行環境**(シェバン・サブシェル・権限)

ここまでは「シェルを快適に使う設定」の話でした。7.2では一歩進んで、**シェルにプログラムを書かせる** 話に入ります。

毎日同じコマンドを順番に何個も打っている ── そんな決まりきった作業は、手順をファイルに書いておいて「これを実行して」と一言で済ませたくなります。その「手順書ファイル」が **シェルスクリプト** です。料理でいえば、毎回レシピを暗唱するのではなく、レシピカードを1枚作っておいて、それを見ながら(あるいは渡すだけで)同じ料理を再現する ── そんなイメージです。一連のコマンドライン作業を **自動化** できるのが最大の利点です。

なお、ここからは「プログラミングらしい」記号や文法が出てきますが、身構える必要はありません。どれも「やりたいこと」と「書き方」をセットで押さえれば理解できます。

### 7.2.1 シェルスクリプトの基礎

#### スクリプトはテキストファイル

シェルスクリプトは、特別な開発ツールがなくても作れます。実行したいコマンドを **テキストファイルに上から順に並べて書くだけ** です。普段コマンドラインで打っている命令を、そのままメモ帳に書き写すイメージです。

たとえば次は、いくつかのコマンドを順番に実行する `lsld` というスクリプトの中身を、`cat` コマンドで表示したものです。

```bash
$ cat lsld
ls -l $1 > lslink
echo "Link Files"
grep '^l' lslink
echo "Directories"
grep '^d' lslink
```

これは「引数で指定した場所の一覧を `lslink` ファイルに書き出し、その中からリンクファイル(`l` で始まる行)とディレクトリ(`d` で始まる行)を抜き出して表示する」という手順書です。一行ずつは、これまで学んだコマンドそのものですね。

#### 4つの実行方法

書いた手順書(スクリプト)を実際に動かす方法は、いくつかあります。ここで重要なのは、方法によって **(1)実行権が必要か** と **(2)どこのシェルで動くか** が異なる点です。試験頻出なので、表で整理して覚えましょう。

![シェルスクリプトの実行方法](images/83_script_exec.png)

| 実行方法 | 実行権 | 動く場所 |
|---|---|---|
| **bash script** | 不要(読み取り権だけでよい) | 子プロセス(サブシェル) |
| **./script** | **必要**(`chmod +x` する) | 子プロセス(サブシェル) |
| **source script** | 不要 | **現在のシェル** |
| **. script** | 不要(`source` の省略形) | **現在のシェル** |

```bash
$ bash lsld           # bashに手順書を渡して読ませる(実行権は不要)
$ chmod a+x lsld      # ファイルに実行権を付ければ
$ ./lsld              # コマンドのように直接実行できる
$ source lsld         # 現在のシェルで実行する
$ . lsld              # source の省略形(カレントディレクトリを表す . とは別物)
```

ここで2つの用語を押さえます。「実行権」とは、第4章で学んだ `x` のパーミッションのこと。`./script` で直接動かすにはこれが必要です。「サブシェル(子プロセス)」とは、いまのシェルが新しく起動するもう1つのシェルのこと ── 親が子を呼び出すイメージです。

> 💡 **`./` が必要な理由**: カレントディレクトリ(今いる場所)は、通常 `PATH`(コマンドを探す場所のリスト)に含まれていません。だから、ただ `lsld` と打ってもシェルは見つけられず、「この場所のこのファイルだよ」と **`./`(=今いるディレクトリ)** を付けて明示する必要があるのです。なお、`PATH` に `.`(カレント)を加えるのは、意図しない場所のプログラムを実行してしまう危険があるため避けます。

> ⚠ **サブシェル vs 現在のシェル(重要)**: `bash` や `./` で実行すると、**新しいbash(子プロセス)が起動** し、そこでスクリプトが動きます。子は仕事を終えると消えるので、**スクリプト内で変数を変えても、終了後の元のシェルには残りません**(子の部屋での変更は親の部屋に反映されない)。一方 `source`(`.`)は **今いるシェル自身で実行** するので、変更がそのまま元のシェルに残ります。設定ファイルの読み込みに `source`(`.`)が使われるのは、「設定をいまのシェルに効かせたい」からです。

#### exec ─ プロセスを置き換える

**exec** コマンドは少し特殊です。新しいプロセスを「子」として作るのではなく、**今のシェルのプロセスそのものを、指定したコマンドに置き換えて** しまいます。乗っ取りのようなもので、元のシェルは戻ってきません(PIDがそのまま引き継がれる)。

```
書式: exec コマンド
```

待機するシェルが不要な場面 ── たとえばX Window Systemでウィンドウマネージャを起動するような場合 ── に使われます。「もうこのシェルには戻らないので、まるごと別プログラムに譲る」という使い方です。

#### スクリプトに渡す引数(位置パラメータ)

スクリプトに「対象」を渡せると、汎用的に使えて便利です。一般のコマンドが `ls /etc` のように引数を取るのと同じで、スクリプトにも実行時に値を渡せます。渡された値は、bashが用意する **特殊な変数** で受け取ります。これを **位置パラメータ** といいます(渡された位置=順番で受け取るため)。

![スクリプトの引数と戻り値](images/84_args_return.png)

| 変数 | 説明 |
|---|---|
| **$0** | スクリプトファイル名(フルパス) |
| **$1** | 1番目の引数(以下 `$2` `$3` ... `$n`) |
| **$#** | 引数の数 |
| **$@** | すべての引数(スペース区切り) |
| **$\*** | すべての引数(区切りは環境変数 IFS で指定されたもの) |

たとえば、引数を表示するだけの `testargs` というスクリプトを `args1` という引数付きで実行すると、次のようになります。

```bash
$ ./testargs args1
./testargs      # $0(スクリプト名そのもの)
args1           # $1(1番目の引数)
                # $2 は空(引数が1つしか渡されていないため)
1               # $#(引数の数=1個)
```

「`$0` は名札、`$1` 以降が中身、`$#` は個数」と整理しておきましょう。

#### 実行結果の戻り値 $?

コマンドは実行を終えるとき、シェルに **「うまくいったか/失敗したか」を数字で報告** します。これが戻り値(終了ステータス)です。ルールはシンプルで、**正常終了なら 0**、**失敗なら 0以外** です。「テストで満点(エラーゼロ)が0点扱い」と考えると逆に思えますが、「エラーの数=0なら成功」とイメージすると納得できます。

この戻り値は特殊変数 **`$?`** に格納され、直前のコマンドが成功したかどうかの判定に使えます。

```bash
$ ls nofile
ls: nofile にアクセスできません: ...
$ echo $?
2               # ls が失敗したので 0以外(この例では2)
$ echo $?
0               # 直前の echo は成功したので 0
```

> ⚠ **戻り値0=正常、0以外=異常** は試験頻出です。注意したいのは、`$?` は **直前のコマンド** の結果しか保持しない点。コマンドを実行するたびに上書きされます(上の例で、2回目の `echo $?` が 0 なのは、1回目の `echo` 自体が成功したから)。`$?` を `if` 文の条件に使えば、「コマンドが成功したら〜する」という分岐が書けます。

#### 📌 試験ポイント

| 問われ方 | 答え |
|---|---|
| 実行権がなくても動かせる実行方法は? | **bash script** / **source** / **.** |
| 実行権が必要な実行方法は? | **./script** |
| 現在のシェルで実行されるのは? | **source** と **.** |
| source の省略形は? | **.**(ドット) |
| 1番目の引数を表す変数は? | **$1** |
| スクリプト名を表す変数は? | **$0** |
| 引数の数を表す変数は? | **$#** |
| すべての引数を表す変数は? | **$@**(または $*) |
| 直前のコマンドの戻り値が入る変数は? | **$?** |
| 正常終了の戻り値は? | **0** |
| プロセスを置き換えるコマンドは? | **exec** |

### 7.2.2 ファイルのチェック

#### test コマンド ─ 条件を判定する

スクリプトで「もしファイルがあれば〜する」のような判断をさせたいとき、その「条件が成り立つか」を調べるのが **test** コマンドです。yes/noを判定する審判のような役割で、条件が成り立てば「真」、成り立たなければ「偽」を返します。

このtestには、まったく同じ働きをする別の書き方 **`[ 条件 ]`** があります。スクリプトでは角カッコの形のほうがよく使われます。

![test コマンド](images/85_test_conditions.png)

```
書式: test 条件文
書式: [ 条件文 ]
```

判定結果は、戻り値と同じ仕組みで表されます。条件が **真(成り立つ)なら 0** を、**偽なら 0以外** を返します。`[ ]` の書き方を使うときは、**`[` の後ろと `]` の前にスペースが必要** です(`[-f file]` ではなく `[ -f file ]`)。これを忘れるとエラーになるので要注意です。

条件式は対象によって3グループに分かれます。

**ファイルのテスト**(ファイルの有無や属性を調べる)

| 条件式 | 真になる条件 |
|---|---|
| **-f ファイル** | (通常の)ファイルがあれば真 |
| **-d ディレクトリ** | ディレクトリがあれば真 |
| **-e ファイル** | ファイルが存在すれば真 |
| **-r / -w / -x ファイル** | 読み取り / 書き込み / 実行が可能なら真 |
| **-s ファイル** | サイズが0より大きければ真 |
| **-L ファイル** | シンボリックリンクなら真 |
| **ファイル1 -nt ファイル2** | ファイル1がファイル2より新しければ真(newer than) |
| **ファイル1 -ot ファイル2** | ファイル1がファイル2より古ければ真(older than) |

**数値のテスト**(数の大小や一致を調べる)

| 条件式 | 真になる条件 |
|---|---|
| **数値1 -eq 数値2** | 等しい(equal) |
| **数値1 -ne 数値2** | 等しくない(not equal) |
| **数値1 -gt 数値2** | より大きい(greater than) |
| **数値1 -ge 数値2** | 以上(greater or equal) |
| **数値1 -lt 数値2** | より小さい(less than) |
| **数値1 -le 数値2** | 以下(less or equal) |

**文字列のテスト・論理結合**(文字列の比較と、条件の組み合わせ)

| 条件式 | 真になる条件 |
|---|---|
| **-n 文字列** | 長さが0より大きければ真(中身がある) |
| **-z 文字列** | 長さが0であれば真(空っぽ) |
| **文字列1 = 文字列2** | 等しければ真 |
| **文字列1 != 文字列2** | 等しくなければ真 |
| **! 条件** | 条件が偽であれば真(否定・ひっくり返す) |
| **条件1 -a 条件2** | 両方が真であれば真(and) |
| **条件1 -o 条件2** | いずれかが真であれば真(or) |

> 💡 **覚え方Hack ─ 数値は英語の略**
> 数値比較は `-eq`(equal) `-gt`(greater than) `-lt`(less than)のように、**英語の頭文字** がそのまま使われています。一方、文字列の等値は記号の `=` `!=` を使います。「数値=英単語の略、文字列=記号」と対比で覚えると混同しません。`-nt`/`-ot` も newer/older than の略です。なお、論理結合の `-a`(and)と `-o`(or)は、シェルオプションの `set -o` の `o` とは無関係なので混同しないように。

#### 📌 試験ポイント

| 問われ方 | 答え |
|---|---|
| 条件を判定するコマンドは? | **test**(または `[ ]`) |
| `[ ]` を使うときの注意は? | **内側にスペースが必要** |
| 真のときの戻り値は? | **0** |
| ファイルの存在を調べる条件式は? | **-e**(通常ファイルは -f) |
| ディレクトリかどうかを調べるのは? | **-d** |
| 数値が等しいことを調べるのは? | **-eq** |
| 数値が大きいことを調べるのは? | **-gt** |
| 文字列が等しいことを調べるのは? | **=** |
| 文字列の長さが0かを調べるのは? | **-z** |
| andで結合するのは? | **-a** |

### 7.2.3 制御構造

シェルスクリプトは、上から下へ順に実行するだけではありません。「条件によって道を分ける(分岐)」「同じ処理を繰り返す(ループ)」といった、流れをコントロールする仕組みがあります。これを **制御構造** といいます。料理のレシピでいう「卵があれば加える(分岐)」「とろみが付くまで混ぜ続ける(繰り返し)」にあたる部分です。

#### if 文 ─ 真/偽で分岐

最も基本的な分岐が **if** 文です。「もし条件が成り立てばAを、そうでなければBを」という、二股の分かれ道です。

![条件分岐 if文とcase文](images/86_if_case.png)

```bash
if 条件式
then
    実行文1      # 条件が真なら、こちらを実行
else
    実行文2      # 条件が偽なら、こちらを実行
fi
```

条件が真なら `then` 以下、偽なら `else` 以下が実行されます。注目してほしいのが終わりの **`fi`**。これは `if` を逆さに綴ったもので、「if文の終わり」を表します(後で出る `case` も同じ発想です)。

```bash
if [ -f testscript ] ; then
    . ./testscript
else
    echo "testscript file not exist"
fi
```

> 💡 `;`(セミコロン)を使うと `if [ -f testscript ] ; then` のように複数行を1行にまとめられます。また、条件式には `test`(`[ ]`)だけでなく **普通のコマンド** も書けます。その場合、**コマンドの戻り値が0(正常終了)なら「真」** とみなされます(7.2.1の戻り値の話がここで効いてきます)。

#### case 文 ─ 値で多分岐

`if` は二股でしたが、「値が1ならコレ、2ならソレ、3ならアレ…」と **たくさんの値で枝分かれ** させたいときは **case** 文が便利です。自動販売機が、押されたボタンに応じて違う商品を出すイメージです。

```bash
case $1 in
  1) echo "January" ;;
  2) echo "February" ;;
  3) echo "March" ;;
  *) echo "other" ;;
esac
```

> ⚠ **case文の文法は頻出**。各分岐の実行文の末尾は **`;;`(セミコロンを2つ)** で締めます。そして全体の終わりは **`esac`**(これも `case` を逆さに綴ったもの)です。最後の `*)` は「どの値にも当てはまらなかった場合」(=その他)を受け持ちます。「`;;` で各分岐を区切り、`esac` で閉じる」をセットで覚えましょう。

#### for 文 ─ リストの数だけ繰り返す

同じ処理を繰り返す「ループ」の代表が **for** 文です。**用意したリストの要素を1つずつ取り出して**、その数だけ処理を繰り返します。「名簿の名前を上から順に1人ずつ呼んでいく」ような動きです。

![繰り返し処理 for文とwhile文](images/87_for_while.png)

```bash
for var in Vine SUSE Gentoo
do
    echo $var Linux
done
```

これは、変数 `var` に `Vine` → `SUSE` → `Gentoo` を順に入れながら、そのつど `echo` を実行します。結果は「Vine Linux」「SUSE Linux」「Gentoo Linux」の3行です。繰り返す処理は `do` と `done` で挟みます。

リストを手で並べる代わりに、**`seq`** コマンドを使うと連続した数値を自動生成できます。

```bash
for i in `seq 10 15`     # 10〜15 の数値を生成して、6回繰り返す
do
    echo $i
done
```

ここで `` `seq 10 15` `` のようにバッククォートで囲むと、その中のコマンドを先に実行し、結果(この場合 `10 11 12 13 14 15`)に置き換わります(第3章のコマンド置換の復習)。

#### while 文 ─ 条件が真の間繰り返す

`for` が「決まった回数」の繰り返しなら、**while** 文は「**条件が成り立っている間ずっと**」の繰り返しです。「お湯が沸くまで待ち続ける」ような、回数を決めずに条件で止めるループです。

```bash
i=1
while [ $i -le 10 ]      # i が10以下の間、繰り返す
do
    echo $i
    let i=i+1            # let で計算。i を1ずつ増やす
done
```

この例は、`i` が1から10になるまで `echo` を繰り返します。`let` は **算術計算を行うコマンド** で、`let i=i+1` で `i` を1増やしています。もしこの「1ずつ増やす」処理を忘れると、条件がいつまでも真のままで **無限ループ** になるので注意です。

#### read ─ 標準入力を受け取る

スクリプト実行中に、**キーボードからの入力を受け取って** 変数に入れたいことがあります。それを行うのが **read** コマンドです。「お名前を入力してください」と尋ねて、打ち込まれた答えを覚えておく、という対話的な処理ができます。

![入力(read)とスクリプトの実行環境](images/88_read_shebang.png)

```bash
echo -n "Who are you? : "    # -n を付けると改行せず、同じ行で入力を待つ
read username                # 入力された文字を変数 username に格納
echo "Hello, $username!"
```

`read` を `while` と組み合わせると、ファイルを **1行ずつ読み込んで処理** できます。次は、ユーザー名が並んだファイルを読んで、自動でユーザーを作成していく例です。

```bash
while read USERNAME
do
    useradd $USERNAME
    echo $USERNAME | passwd --stdin $USERNAME
done < $1                    # $1 で渡したファイルを1行ずつ読む
```

最後の `done < $1` は、「`$1` で渡したファイルの中身を、このループの入力として1行ずつ流し込む」という意味です(リダイレクトの応用)。

### 7.2.4 シェルスクリプトの実行環境

#### シェバン(shebang) ─ 1行目の指定

スクリプトは、書き方がシェルの種類(bash, zsh, sh など)によって少しずつ違います。そこで、「このスクリプトはどのシェルで実行してほしいか」を **1行目** で宣言します。これを **シェバン(shebang)** と呼びます。

```bash
#!/bin/bash
```

この `#!` で始まる1行を先頭に書くことで、スクリプトが指定したシェル(この例では `/bin/bash`)で実行されるようになります。レシピの冒頭に「これは和食の作り方です」と明記して、誤って洋食の手順で作られないようにするイメージです。

> ⚠ **試験頻出 ─ 1行目は `#!/bin/bash`**。先頭の **`#!`** が目印です(この記号の並びがシェバンの語源)。通常のコメントは `#` だけですが、1行目の `#!` だけは特別な意味を持ちます。

#### 実行環境と権限

スクリプトがどんな環境で動くかについて、次の点が試験で問われます。いずれも7.2.1のサブシェルの話と、第4章の権限の話につながっています。

- `bash` や `./` でスクリプトを実行すると、**子プロセス(サブシェル)** が生成され、そこで実行される
- 元のシェルで **export した変数(=環境変数)は、子シェルにも引き継がれる**(伝言板は子も見られる)
- 逆に、**スクリプト内での環境変更は、実行後の元のシェルには反映されない**(子の部屋での変更は親に残らない)
- スクリプトは **実行したユーザーの権限** で動作する
- セキュリティ上の理由から、**スクリプトファイルに設定された SUID / SGID は無視される**

> ⚠ **SUID/SGIDの無視は要注意**。第4章で学んだ通り、実行ファイルにSUIDを付けると「所有者の権限」で動かせます。しかし **シェルスクリプトではこのSUID/SGIDが無視されます**。スクリプトは中身が丸見えで書き換えも容易なため、もし有効だと「root権限を乗っ取る踏み台」にされかねないからです。「スクリプトのSUIDは効かない」と覚えておきましょう。

| 問われ方 | 答え |
|---|---|
| スクリプト1行目に書くシェル指定を何という? | **シェバン(shebang)** |
| bash用スクリプトの1行目は? | **#!/bin/bash** |
| bash/./ で実行したスクリプトはどこで動く? | **子プロセス(サブシェル)** |
| exportした変数は子シェルで有効? | **有効** |
| スクリプトはどの権限で動作する? | **実行したユーザーの権限** |
| スクリプトファイルのSUID/SGIDは? | **無視される** |

#### 📝 ここまでのまとめ

- スクリプト実行: **bash(権限不要・子)/ ./(要実行権・子)/ source・.(現在のシェル)**。**exec** はプロセス置換
- 引数: **$0**(スクリプト名)**$1**(1番目)**$#**(個数)**$@**(全部)。戻り値 **$?**(0=正常、0以外=異常)
- **test**(`[ ]`)で条件判定。ファイル(-f -d -e ...)/ 数値(-eq -gt -lt ...)/ 文字列(= != -z -n)。`[ ]` の内側にスペース必須
- 分岐: **if 〜 fi**(then/else)、**case 〜 esac**(`;;` で区切る)
- 繰り返し: **for 〜 do 〜 done**(リスト/seq)、**while 〜 do 〜 done**(条件が真の間)。**read** で入力
- 実行環境: 1行目 **#!/bin/bash**、サブシェルで動く、**SUID/SGIDは無視される**
